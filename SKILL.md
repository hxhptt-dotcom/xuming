# 续命 (xuming)

## 概述
当收到触发词 `续命` 时，智能体必须立刻停止当前任务。为即将清空上下文或开启新窗口的下一任编程智能体编写“复活存档”文件 `续命.md`。

## 何时使用
- 用户明确输入触发词 `续命`。
- 用户准备执行 `/clear` 或清空上下文、开新窗口。

## 流程指令
一旦触发，请立刻遵循以下规则：

### 1. 允许执行的操作
- 运行只读检查命令以获取当前上下文状态（例如：`pwd`、`git status`、`git diff --stat`）。
- 创建或更新工作区根目录下的 `续命.md`。
- 写完后**立刻停止所有工作**，直接结束，只返回一行简短的物理保存确认文字，以防平台报错，绝不总结，绝不解释，不继续执行任务。

### 2. 绝对禁止的操作
- 禁止继续写代码。
- 禁止继续修 bug。
- 禁止跑会改状态的命令。
- 禁止删除文件。
- 禁止提交 git。
- 禁止美化进度。
- 禁止把没验证的东西写成已完成。
- 禁止把猜测写成事实。
- 禁止废话总结。
- 禁止反问或引导用户。

### 3. `续命.md` 结构规范
必须在工作区根目录下创建 `续命.md`，且只允许使用以下 Markdown 结构：

```markdown
# 续命

## goal
我到底要你干什么。
保留原始目标、后来改过的目标、当前真正要完成的事。

## done
已经做了什么。
写清：
- 看过哪些文件
- 改过哪些文件
- 每个文件大概改了什么
- 做过哪些判断
- 跑过哪些命令
- 哪些验证通过了
- 哪些没验证

## now
当前现场是什么状态。
写清：
- 当前目录
- 当前分支
- 工作区是否有改动
- 正在卡在哪一步
- 最后一件正在做的事
- 当前最重要的文件/代码位置

## blockers
阻塞点。
没有就写：无明确阻塞。
有就写清楚：
- 什么失败了
- 报错是什么
- 为什么还没解决

## hard constraints
下一任 AI 必须遵守的硬限制。
写清：
- 极客协同流：网页端大模型（白嫖额度）先聊透业务流程（不含代码架构），本地端（AGY）直接 `/plan` 精准爆破，禁止废话。
- 只做翻译官：业务流程步骤由老大定死，AI 绝对不能瞎猜业务，只需将蓝图翻译成主流、干净的代码，并选择稳妥的地基架构以便后续迭代。
- 交付自检铁律：交付任何前端项目前，必须用 Playwright 无头浏览器跑自检，并将真机对比截图塞入交付报告中。
- 用户明确要求、项目规则、不能碰的文件、不能回滚的改动。
- 哪些是用户原本的改动、哪些信息只是猜测。

## next step
下一任 AI 接手后第一步该做什么。
只写最小下一步。
必须带验证方式。
```

## 交付与退出
写完 `续命.md` 后，直接停止。不要解释，不要复盘，不要继续执行任务。
